# Разработка RAG-системы для НПА медицинских изделий РК

## Общее описание

Проект представляет собой RAG-систему (Retrieval-Augmented Generation) для ответов на вопросы по нормативно-правовым актам Республики Казахстан в сфере медицинских изделий. Система не обучалась с нуля — использовались готовые предобученные модели, которые настраивались через конфигурацию пайплайна и итеративное улучшение на основе метрик.

---

## Используемые модели

### Эмбеддинг-модель
- **`intfloat/multilingual-e5-large`** (1024 размерности)
- Многоязычная модель на базе E5, обученная на парах вопрос–ответ
- Требует обязательных префиксов: `query:` для запросов, `passage:` для документов
- Используется для индексации документов в Qdrant и для поиска по запросу

### Ре-ранкер (Cross-Encoder)
- **`BAAI/bge-reranker-v2-m3`** (BGE Reranker v2)
- Многоязычный cross-encoder для переранжирования кандидатов
- Выдаёт оценки в диапазоне 0–1 (сигмоид), не логиты
- Запускается на CPU (принудительно) для стабильности на Apple MPS

### LLM
- **Локально:** `qwen2.5:7b` через Ollama (разработка и тестирование)
- **Прод (Railway):** `Gemini 2.5 Flash` через Google GenAI API с fallback на 3 модели

## Архитектура пайплайна

Запрос пользователя проходит через LangGraph StateGraph:

```
Запрос
  → rewrite         (расширение аббревиатур: МИ → медицинские изделия)
  → classify_domain (отказ на запросы о лекарствах)
  → multi_query     (параллельно: 3 варианта запроса + HyDE-пассаж)
  → retrieve        (multi_search: dense + BM25 + RRF для каждого варианта)
  → rerank          (BGE cross-encoder, порог 0.05)
  → check_confidence(отказ если confidence низкий)
  → generate        (топ-5 уникальных по документу чанков → LLM)
  → verify          (проверка: номера статей из ответа должны быть в источниках)
  → ответ
```

### Векторная база
- **Qdrant** — хранение эмбеддингов (3600+ чанков из 27 НПА)
- **BM25** (pymorphy3 + лемматизация) — загружается из Qdrant при старте
- **RRF** (Reciprocal Rank Fusion) — слияние dense и BM25 результатов

## Процесс итеративной настройки

Вместо классического обучения проводилась итеративная оптимизация пайплайна по метрикам на наборе из 28 тестовых вопросов.

### Метрики оценки

| Метрика | Описание |
|---------|----------|
| **Hit Rate@1/3/5** | Как часто нужный документ попадает в топ-N результатов |
| **MRR** | Насколько высоко в выдаче находится правильный ответ |
| **Keyword Coverage** | Доля ключевых слов из эталонного ответа в сгенерированном |
| **Refusal Correctness** | Корректность отказов на нерелевантные вопросы |
| **Verification Failure Rate** | Доля галлюцинированных номеров статей в ответах |
| **Latency p50/p95** | Медианное и 95-й перцентиль времени ответа |

## Ключевые технические решения

### 1. Anchor-чанки (главный прорыв: HR@1 0% → 68%)
Добавление синтетических суммари-чанков для каждого документа:
```
[Решение №46] Этот документ определяет правила регистрации медицинских изделий...
```
Это позволило модели находить нужный документ по общим вопросам типа "каким документом определяется X".

### 2. Структурный чанкинг
Docling конвертирует DOCX → Markdown, затем разбивка по Глава/Статья/Пункт с контекстным заголовком в каждом чанке:
```
[Решение №46] Статья 5. Классификация медицинских изделий...
```

### 3. Multi-query + HyDE
Параллельно генерируются:
- 3 варианта перефразировки запроса через LLM
- Гипотетический документ (HyDE) — ответ, который мог бы быть в базе
Все 4 запроса ищут независимо, результаты сливаются через cross-query RRF.

### 4. Двухэтапный поиск
- **Dense:** cosine similarity через Qdrant (top-100 кандидатов)
- **BM25:** TF-IDF с лемматизацией pymorphy3 (top-100 кандидатов)
- **RRF:** слияние dense+BM25 → top-50 на каждый вариант запроса
- **Cross-query RRF:** слияние результатов всех 4 запросов → top-50 финальных

### 5. Промпт-инжиниринг
- Явный запрет выдумывать номера статей (снизил VFR с 34.6% до 8.3%)
- Правило о частичном ответе: если часть вопроса не покрыта базой — явно указывать это
- Верификация после генерации: номера из ответа должны присутствовать в источниках
